4 · Toast library

Same content in each option: "Approved run #14 · ₹76,959.55 · Q2-FY2025-26". Click the buttons to fire each style. Sonner and react-hot-toast are visually accurate to their library defaults; the Radix custom is a sketch of how we'd build one.

A · Sonner ~5KB · de-facto 2026
+ toast.promise() wraps async engine calls in one line. Rounded card, drop shadow, swipe-dismiss.
− One more 3rd-party design surface.
B · Custom radix-toast ~80 LOC + dep
+ Exact brand match — dark inverted, accent left-bar. Strongest visual identity.
− We re-implement promise toasts. Slower to build, more to own.
C · react-hot-toast ~3KB
+ Smallest, fastest install. Pill shape is friendly.
− Default look is more 2020 than 2026 — needs restyling to match the rest.
My lean: Sonner. The promise-toast pattern fits the engine flow perfectly ("Calculating PVC…" → "Approved · ₹76,959.55") and that one line replaces a useState + conditional render for every async call in the app.

5b · Run-status badges

Approved is the audit-critical state. Draft is malleable. Superseded is historical. Errors block. These will live on every PVC-run row.

5c · Approve action — pattern

Approving a run is irreversible (revisions create superseding runs; the original is locked forever). Treatment options for the trigger button, and the confirm modal I'd build for it:

Loud green — irreversible
Quiet green — restrained
Default primary — no signal of finality
My pick: the loud green, paired with the confirm card to the right. The button's visual loudness signals "this is the moment" — the confirm is what actually prevents misclicks.
